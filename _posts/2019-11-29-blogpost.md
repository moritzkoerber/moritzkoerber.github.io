# Project Background

No matter what business you do: Retaining a customer is almost always cheaper than attracting a new customer. The exact numbers vary from study to study; for example, a survey by [invesp](https://www.invespcro.com/blog/customer-acquisition-retention/) estimated that the cost of acquisition is five times the cost of retention. Thus, any business should care about their customers' churn rate -  the proportion of customers that quit in a certain period. But spending your budget on all of your customers is also not wise. To increase the efficiency of the measures against user churning, it would be great to identify possible churners before they decide to churn. In this blog post, I, therefore, have a look if it is possible to predict user churn in Sparkify, a fictional music streaming app created by Udacity. 

The data set contains the logged activity of its users, such as the song played or the page visited. 

One major reason to cancel a service is that you dislike this particular service. We unfortunately have no ratings, likes or similar indicators in the data set, but it may be possible to infer this from the user behavior. The more a user commits to the service (e.g., by buying a premium plan) or the more a user interacts with a service, the more likely he or she is to like this service. Translated into the data set. My hypothesis is that churn might be predicted by a user's

- the plan (free or paid)
- the intensity of interaction with the app, that is the number of songs played or the duration of each session
- the intensity of interaction with additional features such as adding friends or adding songs to a playlist

Given the fact that churning is a rare event, my evaluation will be based on the f1-score. Due to the large amount of data, I analyze the data set with Spark on multiple nodes realized in an AWS EMR instance.

```pyspark
def calculate_f1score(preds):
    tp = preds.filter((preds.label == 1) & (preds.prediction == 1)).count()
    fp = preds.filter((preds.label == 0) & (preds.prediction == 1)).count()
    tn = preds.filter((preds.label == 0) & (preds.prediction == 0)).count()
    fn = preds.filter((preds.label == 1) & (preds.prediction == 0)).count()
    recall = tp / (tp+fn)
    precision = tp / (tp+fp)
    f1_score = 2*precision*recall / (precision + recall)

    return f1_score, recall, precision
```

# Conclusion and Reflection

The accuracy of this model is 77%, which means that 77% of the users are correctly classified. This sounds like a very good result, but if you compare the f1 score of the churners you can see that the model has a hard time detecting churners and mostly classifies the users as non churners. Since there are 3.39 times more non-churners than churners, the model is right with its classification in many cases, hence the high accuracy. In conclusion, the model does very well in predicting who is not going to churn, because this is true for most of the users, but is not suitable to predict if a user will churn. 

Looking back, the biggest effort in this project probably was feature engineering. It took some time to create aggregate features by user from the supplied time series data. Once this was done, the implementation of the machine learning pipeline was straight forward. Another difficult part was the interpretation of the results. I needed to look at both accuracy and f1 score of both labels to draw a valid conclusion.

# Improvements

The prediction of user churn can probably be improved with additional features. Such features may constitute the time spent on the app, the location or the user agent. Furthermore, it would be interesting to include the chronological structure of the data in the analysis. If a user stops listens to fewer and fewer songs each week, it might be a sign that he/she is about to churn. Also, further external data such as age, income, or gender might help to predict user churn.
